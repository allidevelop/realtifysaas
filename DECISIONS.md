### 2026-06-14 · Этап 0 · PostGIS ставится в схему `gis`, а не `public`
- **Решение:** `CREATE EXTENSION postgis SCHEMA gis` (а не в `public`); `search_path = public, gis`.
- **Альтернативы:** PostGIS в `public` (дефолт `CREATE EXTENSION`).
- **Причина:** Payload (drizzle push) интроспектирует `public` и, увидев чужую таблицу `spatial_ref_sys`, зависает на интерактивном вопросе о переименовании. Размещение PostGIS в `gis` оставляет `public` чистой под Payload — и это ровно разделение схем из ТЗ §6 (public=Payload, gis=PostGIS).

### 2026-06-14 · Этап 0 · Хост-порт Postgres — 5433
- **Решение:** контейнер маппится `5433:5432`; `DATABASE_URL` использует `localhost:5433`.
- **Альтернативы:** стандартный 5432.
- **Причина:** на машине разработки запущена служба `postgresql-x64-17`, занимающая 5432 и перехватывающая TCP-подключения. 5433 убирает конфликт, не трогая существующую установку пользователя.

### 2026-06-14 · Этап 0 · Корневой `.env` загружается в web через `next.config.mjs`
- **Решение:** Next по умолчанию читает `.env` из каталога приложения; секреты лежат в корневом `.env` монорепо (ТЗ §17). В `apps/web/next.config.mjs` подгружаем корневой `.env` через `process.loadEnvFile('../../.env')` (Node ≥20.12).
- **Альтернативы:** дублировать `.env` в каждом приложении.
- **Причина:** единый источник секретов на монорепо; без этого Payload падал с «missing secret key».

### 2026-06-14 · Этап 0 · ESLint-плагины хойстятся через `.npmrc`
- **Решение:** `public-hoist-pattern[]=*eslint*` в корневом `.npmrc`.
- **Причина:** `next lint` через FlatCompat использует legacy-резолюцию плагинов, которая не находит вложенные зависимости `eslint-config-next` под строгим `node_modules` pnpm.

### 2026-06-14 · Коммерческая версия (заменяет трактовку «личное использование»)
- **Решение:** проект — коммерческий продукт (конкурент gisuvecon.com) с монетизацией через подписки и продажу аналитики. «Личное использование» ранее означало лишь «не заказ клиента», а не «некоммерческое». Источник истины — `TZ-geoanalytics-uvecon-v2-commercial.md`.
- **Следствие:** монетизация, биллинг, юр-часть и стратегия данных — обязательные требования, не флаги.
- **Отменяет:** записи ниже про «эквайринг локально не нужен» и «скрейпинг не реализуем» в части их обоснования «личным» проектом (см. уточнения там же).

### 2026-06-14 · Платежи и биллинг — первоклассное требование
- **Решение:** строим рекуррентные подписки (LiqPay/WayForPay/Fondy/Monobank) + безнал по счёту для юрлиц/банков (рахунок-фактура + акт). Доступ к инструментам гейтится по тарифу. Локально пока не настраиваем (этап 5), но это в scope.
- **Альтернативы:** ручная активация без эквайринга.
- **Причина:** доход = подписки и продажи; банки платят только по безналу.
- **Заменяет** прежнее «платежи локально не нужны»: причина была неверной (трактовка проекта как некоммерческого).

### 2026-06-14 · Стратегия данных — на легитимных источниках; накопление своего ряда
- **Решение:** данные объявлений берём из легитимных источников Уровня 1 (официальные API/партнёрства площадок — проверить условия; открытые госреестры; покупка данных) и **накапливаем собственный временной ряд с момента запуска**. Историческую глубину конкурента задним числом не воспроизводим.
- **Скрейпинг чужих сайтов** — НЕ дефолт: юридический риск (ToS, перс. данные, авторские права), решение принимать только с юр-консультацией и фиксировать здесь с оценкой рисков.
- **Альтернативы:** массовый скрейпинг 130+ сайтов «как у конкурента».
- **Причина:** данные — ядро бизнеса и главный юр-риск; легитимный путь устойчивее. Это уточняет (не отменяет цель) прежнюю запись про открытые данные: для коммерции открытых границ мало, нужен реальный поток объявлений.
