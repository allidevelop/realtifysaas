### 2026-06-15 · Этап 3 · Реальные границы АТЕ (geoBoundaries) + привязка по координатам (источник-агностично)
- **Решение:** привязка объявлений к АТЕ — **по координатам** через `ST_Contains` (load.py, level 2), НЕ по гео-словарям конкретного портала. Это работает одинаково для OLX/DOM.RIA/Prozorro (у всех есть lat/lon) и спатиально точно — лучше и проще, чем мапить state/city/district id каждого источника. Реальные границы Украины ставятся одной командой: `import_boundaries --source geoboundaries` (скачивает geoBoundaries ADM1+ADM2, ODbL, через httpx → admin_units; spatial parent-linking). Парсинг свойств обобщён (`_parse_props`: КАТОТТГ-датасеты с укр. названиями/кодами + geoBoundaries shapeName/shapeISO + население). Для real-границ демо-агрегаты по формуле НЕ фабрикуются (флаг `--demo-metrics` опционально) — агрегаты даёт ETL из реальных listings (`aggregate.recompute`).
- **Альтернативы:** (1) гео-словари DOM.RIA (`/dom/states`,`/dom/cities`) с маппингом по названиям — отвергнуто: source-specific, хрупкий name-matching, не нужен при наличии координат. (2) КОАТУУ-фолбэк для объектов без координат (часть аренды/земли Prozorro) — отложено: КОАТУУ (legacy) ≠ КАТОТТГ (new) без кросс-уолка, а основной источник DOM.RIA всегда отдаёт coords.
- **Причина:** ТЗ §9 — реальная choropleth-карта по АТЕ Украины «как у конкурента». geoBoundaries — легитимный (ODbL, атрибуция OSM), документированный, турнкей. Для украинских названий/КАТОТТГ-кодов предпочтителен КАТОТТГ-файл через `--source geojson` (`_parse_props` берёт укр. ключи первыми).
- **Заметка:** geoBoundaries названия могут быть латиницей; для прод-качества (укр. названия + КАТОТТГ) загрузить КАТОТТГ-датасет файлом. Живой `--source geoboundaries` — на машине с сетью (в dev сети нет; парсинг свойств покрыт unit-тестами).

### 2026-06-15 · Этап 3 · OLX Partner API — НЕ источник рынка; основной источник DOM.RIA + «якорь» Prozorro.Sale
- **Решение:** проверено по официальной OpenAPI-спеке (`partner_api.yaml`) и боевому доступу владельца (scope `read write v2`): OLX Partner API — управление **СВОИМИ** объявлениями (`GET /adverts` = «Get user adverts») + сообщения + справочники; **эндпоинта чтения/поиска ЧУЖИХ объявлений и bulk-выгрузки рынка НЕТ**. Значит OLX как источник цен/аналогов для геоаналитики не годится (получение чужих ад'ов = скрейпинг публичной выдачи → Уровень 2 ТЗ §3, юр-риск). OLX-апрув держим как **канал постинга/присутствия**, не как data-source.
- **Источники данных (Уровень 1):** добавлены два коннектора под существующий интерфейс `Source → RawListing`:
  - **`DomRiaSource`** (`apps/engine/etl/sources/domria.py`) — DOM.RIA/DIM.RIA developers API (`developers.ria.com/dom/`): `GET /dom/search` (по группам segment/operation→category/realty_type/operation_type) → `{items:[realty_id],count}`; `GET /dom/info/{id}` → детали (price/currency_type/total_square_meters/lat/lon/publishing_date/beautiful_url). Ключ `DOMRIA_API_KEY` (query). **ОСНОВНОЙ** поток объявлений. Лимит freemium 1000/мес·30/час → коннектор консервативен (`max_pages=1`, `max_items=25`, стоп при 429). **ToS:** обязателен видимый ИНДЕКСИРУЕМЫЙ бэклинк на dom.ria.com (без nofollow/noindex) — реализовать на фронте при показе данных RIA. Коды операций/комерції/землі уточнять через `/dom/options` (заданы документированные дефолты + ENV `DOMRIA_SEARCH_GROUPS`).
  - **`ProzorroSaleSource`** (`apps/engine/etl/sources/prozorro.py`) — Prozorro.Продажі, открытый OCDS-API (`procedure.prozorro.sale/api`, **БЕЗ ключа**): лента `GET /search/byDateModified/{iso}?limit=100` (полные объекты, курсор по `dateModified`+1мс); фильтр недвижимости по `items[].classification` CAV (04*=нерухомість→commercial, 06*=земля→land); operation из `sellingMethod` (lease/rental→rent); цена — реализованная (award/contract) приоритетнее стартовой `value`; площадь из `reProps.totalObjectArea`/`items[].quantity` (HAR→м²×10000); coords из `items[].location`. Это **«якорь» реальных цен сделок** (аукционы гос/комун/арестованного, больше аренда) — НЕ репрезентативный поток жилья.
- **Альтернативы:** скрейпинг OLX/LUN — отвергнут (Уровень 2, юр-решение). LUN/riel без API — только партнёрство.
- **Причина:** ТЗ §3 (поток объявлений с легитимных источников). DOM.RIA — единственный крупный портал UA с публичным read-API листингов; Prozorro — бесплатный верифицированный «якорь» цен (повышает достоверность AVM для банков, ТЗ §10).
- **Проверка:** схемы API проверены живыми запросами (high confidence); мапперы покрыты unit-тестами (`tests/test_etl_sources.py`, 7 кейсов: сегмент/операция/площадь/валюта/coords + нормализация). Живые прогоны коннекторов — на машине владельца (в dev нет внешней сети). DOM.RIA — ready-for-key (апрув запрошен), Prozorro — запускается сразу: `uv run python -m etl.pipeline --source prozorro`.
- **deps:** httpx (уже было). Точные id категорий DOM.RIA office/commercial/land и код operation_type=rent — medium confidence, уточнить через `/dom/options` на боевом ключе.

### 2026-06-15 · Этап 6 · Seats (корп-доступ) — общий entitlement организации, связь через User.organization
- **Решение:** доступ может принадлежать организации (`Entitlement.organization`), а не только пользователю. Членство — однонаправленно через `User.organization` (член → его орг). Резолв доступа org-aware: `getUserEntitlements`/`findActiveEntitlement`/`resolveModuleAccess` ищут `where: { or: [{user}, {organization}] }`, поэтому купленный на орг пакет виден всем её членам (квоты — общий счётчик). Фулфилмент: `applyPaidOrder` выбирает субъект — `order.organization` приоритетнее `order.user`. UI: `/account/organization` (создать орг = стать владельцем+первым членом; добавить/убрать членов по email в пределах `seatsLimit`; владельца убрать нельзя; нельзя «переманить» члена чужой орг). Корп-покупка — чекбокс «для організації» в билинге (виден владельцу орг) → `startCheckout({organizationId})`.
- **Альтернативы:** двусторонний M2M (Organization.members ↔ User) как единственный источник — отверг: для резолва доступа достаточно `User.organization`; `members[]` остаётся для управления/отображения у владельца. Раздача мест копированием entitlement каждому члену — отверг (рассинхрон квот).
- **Причина:** ТЗ §8.2/§16 этап 6 (корпоративные аккаунты с местами). Общий счётчик квоты на орг — ожидаемое B2B-поведение.
- **Проверка (e2e, REST+route):** член с org-entitlement получает полный геопортал (старый период `2026-01` и drill-down `level=2`: 403→200 после выдачи доступа орг); несвязанный freemium-юзер остаётся заблокирован (403). Authz тоггла/покупки — по `ent.user==me` или владению орг.

### 2026-06-15 · Этап 6 · Авто-рекуррент period + dunning/грейс (past_due)
- **Решение:** `PaymentProvider.chargeByToken?(token, amountMinor, …)` (опциональный) — рекуррентное списание по сохранённому токену карты. Mock: детерминированно (токен с подстрокой `fail` → отказ, иначе успех) — для e2e обоих путей. Monobank: `POST /api/merchant/wallet/payment` (`initiationKind: merchant`), токен — из первой токенизированной оплаты (не проверено вживую без боевого токена). Движок `lib/billing/renewals.processRenewals`: кандидаты — period+`autoRenew`+`cardToken`+суммы; **active** продлевается в окне до конца периода (успех → `periodEnd += renewPeriodDays`, чек-квитанция заказом сразу `fulfilled` — без двойного гранта через хук; отказ → `status='past_due'`, `pastDueUntil=now+grace`); **past_due** — грейс истёк (`pastDueUntil≤now`) → `expired`, иначе повтор списания (успех → снова `active`). Триггер — `GET|POST /cron/renewals?secret=` (CRON_SECRET ∥ PAYLOAD_SECRET; на проде — системный cron). Грейс в гейтинге: `resolveModuleAccess` для period допускает `status==='past_due' && pastDueUntil>now`. UI-тоггл авто-продления в билинге (для mock генерит демо-токен `mock_card_<id>`).
- **Альтернативы:** немедленный `expired` при первом отказе (без грейса) — отверг (теряем клиента на разовом сбое карты). Продление через повторный реальный invoice вместо токена — оставлено как путь для не-токенизированных карт.
- **Причина:** ТЗ §11 (рекуррент/отмена/past_due) — раньше были только seam-заглушки.
- **Проверка (e2e, mock+cron+REST):** renew → `+180д` active + receipt-заказ `fulfilled`; отказ → `past_due` (грейс 3д); грейс истёк → `expired`; past_due + рабочая карта → снова `active` (счётчики report: renewed/pastDue/expired/retriedOk).
- **Заметка по тестированию:** server actions (тоггл/покупка/орг) не драйвятся curl (Flight+CSRF) — проверены типчеком/линтом и в браузере прошлых сессий; cron-роут и REST — полноценно curl-тестируемы.

### 2026-06-15 · Этап 3 (данные/ETL) · OLX Partner API + воспроизводимый пересчёт
- **Решение:** ETL-конвейер `apps/engine/etl/`: extract (источник) → transform (дедуп по source+external_id, нормализация валюты в UAH, отсев без площади/цены/координат) → load (upsert в `gis.listings`, привязка к АТЕ через ST_Contains) → `aggregate.recompute` (пересчёт `gis.aggregated_metrics` из listings: IQR-отсев выбросов цены/м², avg+median+count, roll-up область↔район). Источники: `OlxSource` (OAuth2 client_credentials, `GET /adverts`, маппинг advert→listing — **получен апрув OLX**) и `SampleSource` (фикстура OLX-формата для локального e2e без сети/ключей). Оркестратор `etl/pipeline.py` + защищённый триггер engine `POST /api/etl/run?secret=` + ENV `OLX_*`/`ETL_TRIGGER_SECRET`.
- **Причина:** ТЗ §3 (Уровень 1, легитимный источник) + §10.3; DoD «пересчёт воспроизводим» — агрегаты теперь ДЕРИВАТИВ из объявлений, не синтетика. e2e: sample → 10 объявлений (USD→UAH: 41000$→1.68М₴), привязаны к районам, upsert идемпотентен; recompute → 19440 агрегатов; гео/оценка работают.
- **Ограничения/заметки:** из dev нет внешней сети — реальный OLX вызвать нельзя, но коннектор готов (нужны `OLX_CLIENT_ID/SECRET`). Точные id категорий OLX и код атрибута площади (`OLX_AREA_ATTR`) — уточнить по реальному ответу (ENV `OLX_CATEGORY_MAP`). Курсы валют — демо-константы (на проде НБУ на дату). Геокодинг адреса (Nominatim) — пока используем lat/lon из advert. `load_dotenv` в config.py — чтобы os.getenv видел .env (pydantic-settings этого не делает).
- **Накопление ряда (ТЗ §3):** upsert по (source, external_id) + запуск по расписанию (cron на проде) формируют собственный временной ряд.

### 2026-06-15 · Этап 4 (оценка) · Сравнительная оценка на синтетических listings
- **Решение:** движок `app/valuation/comparative.py` — подбор аналогов PostGIS (ST_Within в АТЕ + ST_DWithin 40км, площадь ±35%), веса по схожести (площадь+расстояние), средневзвешенная цена/м², корректировки (торг 0.95, этаж), confidence по числу аналогов и CV. Аналоги генерит `etl/generate_listings.py` (15552, точки ST_GeneratePoints, цены из агрегатов). **Демо-данные** — заменяются реальным потоком на этапе 3.
- **Баг и фикс:** `random()` в нескоррелированном LATERAL кэшировался (все площади одинаковы) → перенесли в CTE-SELECT (volatile per-row).
- **Достоверность для банков (ТЗ §10.1):** возвращаем аналоги, их веса, корректировки и текст методологии (прозрачность, не «чёрный ящик»). Сертификация методики — отдельная бизнес-задача.

### 2026-06-15 · fix · getCurrentUser через cookies()+Authorization JWT
- **Решение:** `lib/auth.getCurrentUser` читает `payload-token` через `cookies()` и передаёт как `Authorization: JWT <token>` в `payload.auth` (а не через `headers()`).
- **Причина:** в server actions заголовки из `headers()` не всегда содержат Cookie (форвардинг через middleware), из-за чего экшены редиректили на login. cookies()-путь надёжен и в RSC, и в actions.
- **Замечание по тестированию:** Next 15 server actions нельзя полноценно драйвить из curl (нужна Flight-кодировка аргументов + Origin); интерактивная проверка форм — в браузере. Связка quota+engine доказана через report-generator (обычный form-POST).

### 2026-06-15 · Этап 2 (геопортал) · Демо-границы (синтетика) + swappable импорт реальных данных
- **Решение:** конвейер геопортала построен на **структурно корректном синтетическом датасете** (полигоны по bbox Украины: 27 «областей» × 4 «района» = 135 АТЕ, КАТОТТГ-подобные коды, синтетические агрегаты за 6 периодов). Скрипт `apps/engine/etl/import_boundaries.py` — **swappable**: `--source geojson --adm1 FILE [--adm2 FILE]` подставляет реальные границы (geoBoundaries/COD-AB/ukrainian_geodata, CC BY 4.0/ODbL) одной командой, parent-связь районов — пространственно (ST_Contains центроида).
- **Причина (конфликт ТЗ §9 с реальностью):** из dev-окружения **нет внешнего сетевого доступа** (Bash-песочница блокирует egress даже без sandbox), скачать реальный GeoJSON нельзя. Синтетика даёт полностью рабочий и проверяемый геопортал сейчас; реальные данные — когда доступен файл/сеть (или их предоставит владелец). Реальный поток объявлений и метрик — этап 3 (там же заменяется синтетика агрегатов).
- **Альтернативы:** блокировать этап до получения данных; ручная авторская разметка координат.
